What is HTML Accessibility?

Note: to understand this documentation, you will first need a basic knowledge of HTML and how it works.

Accessibility is an important principle in the modern practice of web development. Websites should be accessible to the greatest possible number of people, regardless of disabilities and methods for accessing information on the web. Special care must be taken by web developers to properly label and utilize HTML elements in their webpages. It is also important to style things properly so people with blindness and vision-related disabilities can understand and interact with a website.

Semantic Elements

Semantic elements are HTML tags which describe their own purpose to other technologies. Some come with special built-in styling or formatting, while others don't. However, not only can semantic tags improve your SEO, they also allow blind people who use screen reader technology to navigate pages with more ease. A page with poor semantic markup may look cool for seeing users, but it can become a real problem for vision-impaired web users since the screen reader has no signposts to go by and will just read everything on the page at once. Therefore, semantic tags should be used whenever possible.

<div> and <span> may help with some formatting, but they are not semantic. Semantic tags include:

  • <article>
  • <aside>
  • <details>
  • <figcaption>
  • <figure>
  • <footer>
  • <header>
  • <main>
  • <mark>
  • <nav>
  • <section>
  • <summary>
  • <time>

For a more detailed tutorial on the use of these tags, visit the W3Schools tutorial on semantic elements.

The MDN has shown the following example of good semantic markup: 

  <h1>My heading</h1>
  
  <p>This is the first section of my document.</p>
  
  <p>I'll add another paragraph here too.</p>
  
  <ol>
    <li>Here is</li>
    <li>a list for</li>
    <li>you to read</li>
  </ol>
  
  <h2>My subheading</h2>
  
  <p>This is the first subsection of my document. I'd love people to be able to find this content!</p>
  
  <h2>My 2nd subheading</h2>
  
  <p>This is the second subsection of my content. I think is more interesting than the last one.</p>

The following is an example of bad semantic markup: 

  <font size="7">My heading</font>
  <br><br>
  This is the first section of my document.
  <br><br>
  I'll add another paragraph here too.
  <br><br>
  1. Here is
  <br><br>
  2. a list for
  <br><br>
  3. you to read
  <br><br>
  <font size="5">My subheading</font>
  <br><br>
  This is the first subsection of my document. I'd love people to be able to find this content!
  <br><br>
  <font size="5">My 2nd subheading</font>
  <br><br>
  This is the second subsection of my content. I think is more interesting than the last one.

Page Layouts

In earlier days of the internet, HTML tables were used as a way to organize page layouts. This practice is very unhelpful for meeting accessibility requirements, as screen readers read them in a confusing manner. Someone navigating such a website will probably have to enter the table as a separate object and start going through its cells one at a time and it won't make much sense. Now that CSS Flexbox, CSS Grids and other styling and layout options exist and have widespread browser support, it's best practice avoid using HTML tables for page layouts in general, and to only use HTML tables for what they were originally designed for: displaying data and organizing information clearly for users of all abilities. See Data Tables below for more information.

UI Controls

UI controls refer to parts of the website such as buttons, forms, links and other controls with which the user interacts. By default, browsers allow these controls to be selected and focused on using just the keyboard, mostly using the tab key, enter key, arrow keys, etc.

In addition to properly labeling form elements using the <label> tag to assist screen reader technology in helping vision-impaired users navigate forms, making elements keyboard-focusable is important. It is done quite simply by using the correct HTML tags, such as in this example:

        <h1>Links</h1>
  
  <p>This is a link to <a href="https://www.mozilla.org">Mozilla</a>.</p>
  
  <p>Another link, to the <a href="https://developer.mozilla.org">Mozilla Developer Network</a>.</p>
  
  <h2>Buttons</h2>
  
  <p>
    <button data-message="This is from the first button">Click me!</button>
    <button data-message="This is from the second button">Click me too!</button>
    <button data-message="This is from the third button">And me!</button>
  </p>
  
  <h2>Form</h2>
  
  <form>
    <div>
      <label for="name">Fill in your name:</label>
      <input type="text" id="name" name="name">
    </div>
    <div>
      <label for="age">Enter your age:</label>
      <input type="text" id="age" name="age">
    </div>
    <div>
      <label for="mood">Choose your mood:</label>
      <select id="mood" name="mood">
        <option>Happy</option>
        <option>Sad</option>
        <option>Angry</option>
        <option>Worried</option>
      </select>
    </div>
  </form>

While this is quite simple, occasionally there are still developers who don't use the proper elements. For example:

  <div data-message="This is from the first button">Click me!</div>
  <div data-message="This is from the second button">Click me too!</div>
  <div data-message="This is from the third button">And me!</div>

While there are some workarounds, such as using the attribute of tabindex="0" and adding some Javascript code, these can still lead to cumbersome problems and should be avoided in the first place if at all possible. Visit MDN's article on the topic for a more detailed explanation of how to add in keyboard accessibility.

Image Accessibility

Images cannot be seen well by visually impaired people, which means it is important to use the alt attribute to assist screen readers in making them accessible. Here are two examples of code:

  <img src="dinosaur.png">
  
  <img src="dinosaur.png" alt="A red Tyrannosaurus Rex: A two legged dinosaur standing upright like a human, with small arms, and a large head with lots of sharp teeth.">

The first image is not helpful at all. A screen reader will read the file name, which may provide no context at all. The second example, however, will provide the user with a much richer and more accessible experience as the screen reader will read the alternate text to them.

In cases where the image is purely for visual decoration and is not important for an accessible understanding of the site content, it is still important to use empty alt attributes so that the screen reader can respond appropriately. You can also use the aria role attribute role="presentation" so the screen reader does not read out a long, confusing file name and treats the image appropriately. 

  <h3>
    <img src="article-icon.png" alt="">
    Tyrannosaurus Rex: the king of the dinosaurs
  </h3>

Data Tables

Proper elements should be used when making data tables in order to allow screen readers to provide users with information that is more easily understood and navigated. Below is an example of code for a simple data table that should be improved upon in this area:

   <table>
    <tr>
      <td>Name</td>
      <td>Age</td>
      <td>Gender</td>
    </tr>
    <tr>
      <td>Gabriel</td>
      <td>13</td>
      <td>Male</td>
    </tr>
    <tr>
      <td>Elva</td>
      <td>8</td>
      <td>Female</td>
    </tr>
    <tr>
      <td>Freida</td>
      <td>5</td>
      <td>Female</td>
    </tr>
  </table>

While this table may appear normal visually, it's not very helpful for people using a screen reader because the technology has no way of knowing how to group the rows and columns together. It's much better to be more specific and label the table properly. Some examples of how to do this are:

  • Use <th>elements to label table header cells. Use the scope attribute to specify whether they are headers for rows or columns, allowing users to experience the entire group of data at once and more coherently.
  • Use <thead> to group the entire header row
  • Use <tfoot> when appropriate
  • Use <tbody> and <colgroup> to group rows and columns if appropriate
  • Use the <caption> element or the <table> summary attribute to act as an alternate text, allowing screen readers to provide a quick summary for users of what the table contains.

MDN's Github page has an example of better, more accessible table markup.

Summary

HTML accessibility is important in order for your website to reach the largest audience possible, regardless of ability or method of web access. To sum up, here are some quick tips:

  • Use semantic elements wherever possible.
  • Do not use HTML tables for page layouts. Instead, use modern, responsive CSS models.
  • Make sure forms and UI controls are both properly labeled and keyboard-focusable.
  • Always add alternate text to images, even if you have to leave the alt attribute empty. Specify the image's role attribute if necessary.
  • Be specific when organizing data tables so that screen readers can group rows and columns together and recognize their headers. Add captions to tables as well.
